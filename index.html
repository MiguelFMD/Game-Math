Tema 1: Análisis y costes de algoritmos

1. ¿Qué es la eficiencia?

En programación, que un código funcione no es suficiente; debe ser eficiente. Esto significa resolver el problema consumiendo los mínimos recursos posibles.

A menudo hay que elegir: para ganar tiempo (velocidad), solemos gastar más memoria (índices, tablas auxiliares), y viceversa.

Tipos de Costes Principales
  • Coste Temporal: Cuánto tiempo tarda el procesador (CPU) en ejecutarlo. Es el más crítico habitualmente.
  • Coste Espacial: Cuánta memoria (RAM) extra necesita para funcionar.

2. ¿Cómo medimos el coste?

No medimos la eficiencia en segundos porque eso depende de la potencia de tu ordenador o del lenguaje usado. Para medir de forma científica usamos:

  • Pasos: Una operación básica (sumar, asignar variable, comparar if) cuenta como 1 paso.
  • Talla del problema ($n$): Es el tamaño de los datos de entrada. Si ordenas una lista de 100 números, $n=100$.

El objetivo es encontrar una fórmula (función) que nos diga cuántos pasos da el algoritmo según crece $n$.

Ejemplo visual: Calcular $n^2$

Imagina tres formas de calcular el cuadrado de un número $n$:

  • Forma A (Directa): $m = n * n$. Tarda 1 paso, sin importar lo grande que sea $n$.
  • Forma B (Sumas): Sumar $n$ veces (ej: $5^2 = 5+5+5+5+5$). Si $n$ es 1000, hace 1000 sumas.
  • Forma C (Contar): Contar de uno en uno hasta llegar al resultado. Si $n$ es 1000, cuenta hasta 1.000.000 ($1000 \times 1000$).
La clave del crecimiento

Para $n$ pequeño no notarás la diferencia. Pero si $n$ es 1 millón: la Forma A es instantánea, la B tarda un poco, y la C podría tardar años. Lo importante no es el tiempo exacto, sino cómo crece el tiempo al aumentar los datos.

3. El Peor Caso es el que importa

Como el tiempo puede variar según los datos (ej: ordenar una lista que ya está ordenada es rápido), nos ponemos siempre en tres situaciones:

  • Mejor Caso: Tienes suerte y los datos son ideales. Poco útil en la práctica.
  • Peor Caso: Todo sale mal (ej: buscas un dato que está al final). Este es el que usamos en ingeniería, porque nos garantiza que el programa nunca tardará más que esto.
  • Caso Promedio: Lo que suele ocurrir normalmente.

4. La Notación "O Grande" ($O$)

Es la etiqueta que le ponemos al algoritmo para clasificarlo. Nos dice el "techo" de su crecimiento. Aquí tienes el ranking de eficiencia, de mejor a peor:

  • $O(1)$ Constante: Instantáneo. No depende de los datos (ej: acceder a un array).
  • $O(\log n)$ Logarítmica: Rapidísimo. Si duplicas los datos, solo añades un paso más (ej: Búsqueda Binaria).
  • $O(n)$ Lineal: Proporcional. Si doblas los datos, doblas el tiempo (ej: leer una lista).
  • $O(n \log n)$ Cuasilineal: Muy rápido. El estándar para ordenar datos (ej: Quicksort, Mergesort).
  • $O(n^2)$ Cuadrática: Lento. Bucles anidados. Aceptable solo para pocos datos (ej: Burbuja).
  • $O(2^n)$ Exponencial: Inviable. El tiempo se dispara imposiblemente rápido (ej: contraseñas por fuerza bruta).

5. Pila de Llamadas y Costes (Coste Espacial)

El Coste Espacial (o complejidad espacial) mide cuánta memoria necesita un algoritmo para ejecutarse, en función del tamaño de la entrada ($n$). No contamos el espacio que ocupan los datos originales (el vector de entrada), sino la memoria extra que necesitamos para trabajar.

¿Qué incluye exactamente ese "espacio extra"?

  1. Variables Simples: Contadores ($i, j$), acumuladores (suma), variables temporales (temp). Suelen ocupar espacio constante $O(1)$.
  2. Estructuras Auxiliares: Si creas un nuevo vector, una matriz o una lista para guardar resultados intermedios. Si copias un vector de tamaño $n$, gastas $O(n)$ de memoria.
  3. Llamadas Recursivas (La Memoria Oculta): Este es el punto más crítico y difícil de ver. Cada vez que una función se llama a sí misma, el ordenador debe guardar el estado actual (variables locales, punto de retorno) en una zona de memoria llamada Pila de Llamadas (Stack).

Ejemplo: La Secuencia de Fibonacci

Comparar la versión recursiva e iterativa de Fibonacci ilustra perfectamente esta diferencia:

// Fibonacci Iterativo: Coste Temporal O(n) | Espacial O(1)
int FibonacciIterativo(int n) {
    int a = 0, b = 1, temp; // Usa variables simples
    for (int i = 2; i <= n; i++) {
        temp = a + b;
        a = b;
        b = temp;
    }
    return b;
}

// Fibonacci Recursivo: Coste Temporal O(2^n) | Espacial O(n)
int FibonacciRecursivo(int n) {
    if (n <= 1) return n;
    return FibonacciRecursivo(n - 1) + FibonacciRecursivo(n - 2); // Repite cálculos millones de veces
}
¡Cuidado con el Stack Overflow!

En el Fibonacci recursivo, aunque no declaras vectores, para calcular $fib(n)$ necesitas llegar hasta $fib(1)$. Esto crea una cadena de llamadas abiertas de profundidad $n$ en la pila. Si $n$ es muy grande, puedes agotar la memoria y provocar un error de Stack Overflow.

6. Algoritmos Famosos: Tabla de Costes Temporales y Espaciales

A continuación, se detallan los costes. Recuerda que el Coste Espacial mostrado es la memoria auxiliar (extra); no contamos el espacio del propio vector de datos, que siempre es $n$.

Algoritmo Tiempo (Mejor) Tiempo (Promedio) Tiempo (Peor) Espacio (Auxiliar) Notas importantes
Búsqueda Lineal $O(1)$ $O(n)$ $O(n)$ $O(1)$ No requiere memoria extra.
Búsqueda Binaria $O(1)$ $O(\log n)$ $O(\log n)$ $O(1)$ Versión iterativa. Muy eficiente en espacio y tiempo.
Burbuja $O(n)$ $O(n^2)$ $O(n^2)$ $O(1)$ Ordena "in-place", por eso gasta poco espacio.
Inserción $O(n)$ $O(n^2)$ $O(n^2)$ $O(1)$ Muy eficiente ($O(n)$ tiempo) si la lista ya está casi ordenada.
Selección $O(n^2)$ $O(n^2)$ $O(n^2)$ $O(1)$ Ineficiente en tiempo, pero muy bajo consumo de memoria.
Quicksort $O(n \log n)$ $O(n \log n)$ $O(n^2)$ $O(\log n)$ Gasta espacio en la pila debido a la recursividad.

Resumen de uso:

  • Para ahorrar memoria ($O(1)$): Usa algoritmos iterativos como Inserción (si son pocos datos) o Heapsort (si son muchos, aunque es más complejo de programar).
  • Para velocidad ($O(n \log n)$): Usa Quicksort. Aunque gasta un poco de memoria en la pila ($O(\log n)$), es el más rápido en la práctica.
  • ¡Cuidado con la recursividad! Algoritmos como Quicksort pueden tener un coste espacial oculto en el peor caso ($O(n)$) si la partición de datos es mala, pudiendo desbordar la pila.
